# 智能金融顾问系统（Multi-Agent 方向）工作流程文档

## 1. 文档目的
本文档用于描述智能金融顾问系统从需求进入、Agent 协作、结果生成到测试部署和持续迭代的完整工作流程，帮助产品、算法、工程、测试、合规团队在同一流程框架下协同执行。

## 2. 流程总览
项目整体遵循“需求拆解 -> 架构搭建 -> 模块开发 -> 集成部署 -> 测试优化”的推进方式，线上运行遵循“用户输入 -> 任务拆解 -> Agent 协作 -> 合规过滤 -> 输出结果 -> 监控反馈”的闭环流程。

## 3. 项目实施流程

### 3.1 阶段一：项目启动与需求拆解
目标：明确业务问题、监管边界、核心指标和技术方向。

工作内容：
- 联合产品、业务、合规团队梳理典型场景和高优先级问题。
- 明确核心指标，包括时延、并发、自动化率、准确率、合规率。
- 识别监管红线，例如外汇管制、产品适格性、风险匹配要求。
- 形成最小可行方案和 PoC 验证范围。

输出物：
- 需求文档。
- 场景清单与优先级。
- 技术选型初稿。
- PoC 验证计划。

### 3.2 阶段二：知识库与工具链建设
目标：完成 Agent 运行所依赖的数据底座和接口能力。

工作内容：
- 解析金融产品说明书、法规文档、市场报告等数据源。
- 清洗与标准化结构化、半结构化、非结构化数据。
- 构建 Milvus 向量库和 Neo4j 知识图谱。
- 接入市场数据 API、用户资产接口、合规规则接口。
- 建立缓存、重试、权限校验和失败 fallback 机制。

输出物：
- 文档解析与入库流程。
- 向量库与图谱结构设计。
- 工具调用封装服务。
- 数据同步策略与更新规则。

### 3.3 阶段三：Multi-Agent 核心模块开发
目标：完成主 Agent、子 Agent、过滤 Agent 的能力建设。

工作内容：
- 开发主 Agent 的意图识别、参数抽取、任务拆解和调度逻辑。
- 开发风险评估 Agent 的风险打分与解释能力。
- 开发投资推荐 Agent 的筛选、排序、组合理财能力。
- 开发合规检查 Agent 的规则校验和驳回反馈能力。
- 开发内容过滤 Agent 的敏感信息过滤与可解释性校验能力。
- 定义统一状态结构和 JSON 输出协议。

输出物：
- Agent 职责说明。
- 状态字段与协议文档。
- Prompt、规则、模型配置。
- 单模块测试用例。

### 3.4 阶段四：系统集成与部署
目标：完成服务组装、容器化部署、高并发优化与监控上线。

工作内容：
- 用 LangGraph 编排主流程，用 AutoGen 处理多 Agent 协作细节。
- 通过 FastAPI 对外提供统一服务接口。
- 采用 Docker 封装各模块，使用 K8s 完成部署与扩缩容。
- 通过 vLLM 进行模型部署和推理性能优化。
- 建立 LangFuse 与 Grafana 的监控和实验管理。

输出物：
- 集成后的服务链路。
- 容器化镜像与部署配置。
- 监控看板与告警规则。
- 上线前联调报告。

### 3.5 阶段五：测试与迭代优化
目标：完成上线前验证和上线后持续优化闭环。

工作内容：
- 功能测试覆盖典型投资咨询场景。
- 性能测试验证高并发与大流量能力。
- 合规测试验证规则完整性与输出合法性。
- 上线后持续分析日志、监控与用户反馈。
- 持续优化 Prompt、权重、检索逻辑和流程策略。

输出物：
- 功能测试报告。
- 性能压测报告。
- 合规专项测试报告。
- 迭代优化清单。

## 4. 用户侧业务流程

### 4.1 主流程
1. 用户通过 App 或 Web 输入投资需求。
2. 主 Agent 识别用户意图并抽取关键参数。
3. 若信息不完整，主 Agent 发起补充提问。
4. 主 Agent 生成任务链路并启动风险评估 Agent。
5. 风险评估 Agent 输出风险等级和适配产品范围。
6. 投资推荐 Agent 获取实时市场数据和产品知识，生成候选组合。
7. 合规检查 Agent 校验外汇额度、风险匹配和产品适格性。
8. 内容过滤 Agent 完成敏感信息过滤和解释性检查。
9. 系统生成最终建议方案并返回给用户。
10. 用户可继续追问，系统基于上下文继续回答。

### 4.2 输出结构
最终输出建议应至少包含以下内容：

- 推荐方案。
- 配置比例。
- 推荐依据。
- 风险提示。
- 数据来源与政策依据。

## 5. Agent 协作流程

### 5.1 协作原则
- 主 Agent 负责决策与调度，不直接执行业务计算。
- 子 Agent 各自只处理单一职责范围内的任务。
- 所有中间结果写入共享状态，避免直接耦合。
- 合规优先于收益推荐，过滤 Agent 作为最终兜底。

### 5.2 状态流转
1. 主 Agent 读取用户输入并写入初始状态。
2. 风险评估 Agent 读取用户信息并写回风险结果。
3. 投资推荐 Agent 基于风险结果和工具数据生成候选组合。
4. 合规检查 Agent 对候选组合执行校验并写回结论。
5. 内容过滤 Agent 基于最终内容执行脱敏和解释性检查。
6. 主 Agent 汇总各 Agent 结果，输出最终响应。

### 5.3 条件分支
- 若缺少本金、投资期限、风险偏好等关键信息，则进入补参分支。
- 若用户属于高净值场景，则增加资质校验分支。
- 若需求只是简单信息查询，则走流程剪枝，直接返回知识检索结果。
- 若合规不通过，则返回投资推荐 Agent 进行重新筛选。

## 6. 工具调用流程

### 6.1 市场数据 API
- 调用主体：投资推荐 Agent。
- 调用条件：涉及实时净值、汇率、收益率或缓存超时。
- 处理策略：优先实时查询，失败时回退到 1 小时内缓存。

### 6.2 用户资产查询接口
- 调用主体：风险评估 Agent、合规检查 Agent。
- 调用条件：用户已授权且任务依赖资产、持仓、额度信息。
- 处理策略：先做权限验证，再做字段校验与格式转换。

### 6.3 合规规则库接口
- 调用主体：合规检查 Agent。
- 调用条件：涉及海外投资、高风险产品、适格性判断。
- 处理策略：获取规则后执行本地规则匹配，并记录依据条款。

### 6.4 工具失败兜底
1. 首次失败后进行重试，最多 3 次。
2. 仍失败则尝试 fallback 数据。
3. fallback 不可用则返回异常提示并记录日志。
4. 连续失败达到阈值时触发 Grafana 告警。

## 7. 异常处理流程

### 7.1 信息缺失
- 主 Agent 识别缺失字段。
- 发起补充提问。
- 若用户拒绝提供关键信息，则停止自动建议并转人工。

### 7.2 协作失败
- 若前序 Agent 未返回结果，则阻断后续流程。
- 若存在死循环风险，则设置最大循环次数，超限后中断。
- 若状态传递不完整，则记录错误并进入排查链路。

### 7.3 推荐不符合预期
- 回查主 Agent 的参数提取结果。
- 回查风险评估的打分与权重。
- 回查推荐 Agent 的产品筛选逻辑。
- 回查合规 Agent 是否改写了推荐结果。

## 8. 团队协作流程
1. 业务或合规团队提出新增需求或规则变更。
2. 产品团队完成场景梳理与优先级定义。
3. 算法团队更新 Agent 逻辑、知识库或模型配置。
4. 工程团队完成接口、服务、部署和监控改造。
5. 测试团队执行功能、性能、合规专项验证。
6. 合规团队进行最终审核。
7. 工程团队发布上线并监控运行效果。
8. 上线后一周内进行跨团队复盘，确定下一轮优化项。

## 9. 监控与反馈闭环

### 9.1 监控指标
- 任务拆解准确率。
- 风险匹配准确率。
- 合规通过率。
- 敏感信息泄露率。
- 响应时延。
- QPS。
- 工具调用成功率。

### 9.2 反馈闭环
1. 通过 LangFuse 收集日志、轨迹、Prompt 和用户反馈。
2. 通过 Grafana 观察异常、性能和业务波动。
3. 定位问题归属到数据、模型、流程或接口层。
4. 制定修复策略并执行回归验证。
5. 记录版本变更与优化结果。

## 10. 上线后持续优化流程
- 根据监控数据调整风险评估指标权重。
- 根据用户反馈优化 Prompt 和追问逻辑。
- 根据合规政策变化更新规则库和审核模板。
- 根据市场波动调整数据同步频率与推荐策略。
- 对高风险建议增加复核机制。
