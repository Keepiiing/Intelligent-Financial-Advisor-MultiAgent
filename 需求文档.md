# 智能金融顾问系统（Multi-Agent 方向）需求文档

## 1. 文档目的
本文档用于明确智能金融顾问系统的建设目标、业务范围、核心能力、功能需求、非功能需求及验收指标，作为产品、算法、工程、测试、合规团队协同推进项目的统一依据。

## 2. 项目背景
传统金融咨询在投资建议场景中存在以下问题：

- 响应慢，依赖人工顾问，通常需要 1 至 2 天反馈。
- 个性化能力不足，难以针对用户画像输出差异化建议。
- 风险评估流程割裂，缺少统一的自动化决策链路。
- 合规校验分散，难以保证输出稳定满足监管要求。

本项目面向金融科技投资顾问场景，建设一个以 Multi-Agent 为核心的智能金融顾问系统，输出实时、个性化、合规、可解释的投资建议。

## 3. 项目目标

### 3.1 业务目标
- 为用户提供实时、个性化、合规的智能投资建议。
- 支持海外基金投资、投资组合优化等复杂金融决策场景。
- 降低对人工顾问的依赖，提升自动化建议处理率。
- 建立可审计、可追溯、可持续优化的智能顾问能力。

### 3.2 量化目标
- 响应时延小于 1 秒，目标稳定在 800ms 以内。
- 每日支持 8 万次以上查询。
- 支持 100 名以上用户同时在线并发。
- 自动化建议处理率不低于 60%。
- 风险评估准确率达到 88%。
- 合规通过率达到 100%。
- 敏感信息泄露率为 0。

## 4. 项目范围

### 4.1 范围内
- 用户投资需求理解与参数提取。
- 多 Agent 协作的投资建议生成。
- 风险评估、产品推荐、合规检查、内容过滤。
- 金融知识库与外部工具链接入。
- 监控、审计、测试、部署及迭代优化能力建设。

### 4.2 范围外
- 直接执行交易下单。
- 替代人工进行高风险或特殊资产业务最终审批。
- 未授权用户的资产查询。
- 不具备监管依据的收益承诺或保本承诺输出。

## 5. 用户与角色

### 5.1 外部用户
- 普通投资用户：提交投资需求并接收建议方案。
- 高净值用户：需要额外资质判断及专属产品推荐。

### 5.2 内部角色
- 产品团队：定义需求、梳理场景、制定验收口径。
- 金融业务团队：提供产品规则、投资逻辑、业务约束。
- 合规团队：提供政策标准、审核规则、审计要求。
- 算法团队：设计 Agent 策略、模型能力、知识检索逻辑。
- 工程团队：建设服务、接口、部署、监控与稳定性能力。
- 测试团队：负责功能、性能、合规专项测试。

## 6. 核心业务场景
- 海外基金投资建议。
- 投资组合优化建议。
- 基于用户画像的个性化产品推荐。
- 高净值用户专属产品推荐。
- 用户追问式交互，例如最大回撤、风险来源、推荐依据等。

## 7. 总体方案概述
系统采用“1 主 3 子 + 1 过滤”的 Multi-Agent 架构：

- 主 Agent：负责意图识别、参数提取、任务拆解和流程调度。
- 风险评估 Agent：负责评估用户风险等级和适配产品类型。
- 投资推荐 Agent：负责筛选产品并生成候选组合方案。
- 合规检查 Agent：负责校验外汇管制、风险匹配和产品适格性。
- 内容过滤 Agent：负责敏感信息过滤和可解释性校验。

支撑层由知识库、外部工具链和监控体系组成，保障各 Agent 的协同执行。

## 8. 功能需求

### 8.1 用户需求接入
- 系统应支持用户通过 App、Web 等渠道提交投资需求。
- 系统应识别用户输入中的本金、风险偏好、投资期限、投资标的、地域偏好等核心参数。
- 当关键信息缺失时，系统应主动发起补充询问。

### 8.2 任务拆解与流程编排
- 系统应将复杂需求拆解为风险评估、产品筛选、合规校验、结果生成等子任务。
- 系统应支持线性流程与条件分支流程。
- 系统应基于共享状态传递上下文与任务结果，避免信息丢失。

### 8.3 风险评估
- 系统应基于风险测试得分、资产规模、投资期限、过往投资行为、流动性需求等指标进行风险评估。
- 系统应输出风险等级、风险说明及适配产品类型。
- 系统应支持主观偏好与客观数据交叉验证，不一致时触发确认机制。

### 8.4 投资推荐
- 系统应结合风险评估结果、市场实时数据、用户画像和产品知识库生成推荐方案。
- 系统应至少输出 3 套组合方案：保守、均衡、进取。
- 系统应在方案中体现产品名称、配置比例、推荐依据和风险提示。

### 8.5 合规检查
- 系统应校验海外投资外汇额度限制。
- 系统应校验推荐产品风险是否超过用户承受能力。
- 系统应校验产品适格性，例如高风险产品是否仅面向符合资产门槛的用户。
- 对于不合规结果，系统应返回驳回原因及修正建议。

### 8.6 内容过滤与可解释性
- 系统应过滤身份证号、完整资产金额、银行卡号等敏感信息。
- 系统输出必须包含推荐依据、风险提示和来源引用。
- 系统不得输出保本承诺、明确收益保证等违规表述。

### 8.7 外部工具调用
- 系统应接入市场数据 API，获取基金净值、汇率、市场动态。
- 系统应接入用户资产查询接口，获取持仓、可用资金、外汇额度。
- 系统应接入合规规则库接口，获取产品适格性和监管规则。
- 工具调用必须包含权限校验、重试机制、缓存策略和失败兜底逻辑。

### 8.8 知识库能力
- 系统应支持解析产品说明书、合规法规、市场报告等多类型文档。
- 系统应将非结构化知识写入 Milvus，将结构化关系写入 Neo4j。
- 系统应支持定时同步与主动触发的知识更新机制。

### 8.9 追问与会话延续
- 用户收到方案后，系统应支持围绕当前推荐结果继续追问。
- 系统应基于上下文回答诸如风险来源、最大回撤、替代方案等问题。

### 8.10 异常与转人工
- 若用户拒绝提供关键信息，系统应停止自动建议并转接人工顾问。
- 若工具调用失败且兜底方案不可用，系统应返回明确提示并记录异常。
- 若流程重复调用超过阈值，系统应中断并进入异常处理流程。

## 9. 非功能需求

### 9.1 性能要求
- 高并发场景下响应时延应小于 1 秒。
- 单实例 QPS 目标为 2000 以上。
- 系统应支持动态扩缩容。

### 9.2 稳定性要求
- 工具调用应具备自动重试和 fallback 机制。
- 连续失败应触发监控告警。
- 关键流程应支持状态恢复与错误追踪。

### 9.3 安全与隐私要求
- 敏感数据应加密存储和传输。
- Agent 间共享状态中的敏感字段应进行权限控制。
- 输出结果必须经过敏感信息扫描。

### 9.4 合规要求
- 全流程必须嵌入合规校验节点。
- 所有建议记录、校验依据、时间戳应可追溯。
- 输出内容必须满足金融监管对适当性、可解释性、审计性的要求。

### 9.5 可维护性要求
- Agent 输出协议应统一为结构化格式。
- 各 Agent 应职责清晰、可独立迭代、可复用。
- 监控、日志、实验版本应可统一管理。

## 10. 数据与接口需求

### 10.1 输入数据
- 用户自然语言需求。
- 用户授权后的资产与持仓数据。
- 风险测试问卷结果。
- 市场行情与基金净值数据。
- 产品知识文档与合规法规文档。

### 10.2 输出数据
- 投资建议方案。
- 风险等级及适配说明。
- 合规校验结论。
- 推荐依据、风险提示、来源引用。
- 审计记录与监控指标。

## 11. 技术与实现约束
- 后端建议采用 Python + FastAPI。
- Agent 编排建议采用 LangGraph + AutoGen。
- 检索与知识管理建议采用 LlamaIndex、Haystack、Milvus、Neo4j。
- 模型推理建议采用 vLLM 部署。
- 容器化与编排建议采用 Docker + K8s。
- 监控与实验管理建议采用 LangFuse + Grafana。

## 12. 验收标准

### 12.1 功能验收
- 能完整跑通“需求输入 -> 风险评估 -> 投资推荐 -> 合规校验 -> 内容过滤 -> 输出建议”主链路。
- 能处理用户追问、缺参补全、异常兜底与人工转接。
- 能接入并调用市场数据、资产查询、合规规则三类外部接口。

### 12.2 指标验收
- 响应性能、自动化处理率、风险评估准确率、合规通过率达到既定目标。
- 在压测环境中满足 8 万次以上日查询量与 100+ 并发要求。

### 12.3 合规验收
- 通过合规团队专项审核。
- 敏感信息脱敏、引用来源、风险提示满足规范。
- 推荐记录可查询、可审计、可回溯。

## 13. 风险与关注点
- 市场数据时效性直接影响推荐有效性，需要强化数据新鲜度校验。
- 用户自报风险偏好可能失真，需要交叉验证和确认机制。
- Agent 协作可能引发循环或冲突，需要状态机和阈值保护。
- 金融合规规则变动快，需要建立快速更新与回归验证机制。

## 14. 后续建议
- 增加高风险建议复核 Agent。
- 增加情绪识别能力，优化交互体验。
- 引入突发政策与市场风险知识库。
- 逐步扩展到保险推荐、综合理财规划等场景。
